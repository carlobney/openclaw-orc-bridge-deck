Bridge Plan

OpenClaw + Orc

Tap the interactions already happening, feed them into Grain, and let Orc earn its way into the loop.

Thesis

Start passive. Prove value. Then move workflow authority.

The first bridge should not replace the live agent path. It should observe, normalize, and create durable events from real OpenClaw traffic.

What exists

The pieces are already on the machine.

OpenClaw

Message hooks

Internal and plugin hooks can observe received, preprocessed, sent, session, gateway, cron, and tool events.

Grain

Event backbone

CQRS, event sourcing, read models, todo processors, and periodic tasks give the bridge a durable substrate.

Orc

Workflow brain

Behavior-tree workflows can evaluate events and decide when deeper handling is worth invoking.

First tap

A real hook is already in place.

  • hooks/orc-interaction-tap/ is enabled as an internal OpenClaw hook.
  • It listens to message:received, message:preprocessed, and message:sent.
  • It writes bounded sanitized records to data/orc-interaction-tap.jsonl.
  • That JSONL append is intentionally replaceable by a Grain event append.
Architecture

The bridge path stays narrow.

1

OpenClaw hook

Observe message lifecycle events without changing response behavior.

2

Grain event

Append normalized events into SQLite-backed event storage.

3

Orc workflow

Classify, enrich, route, or recommend next actions from read models.

4

OpenClaw delivery

Post back only when the workflow produces an explicit visible result.

Data contract

The event shape should be boring.

; Grain event names
:openclaw/message-received
:openclaw/message-preprocessed
:openclaw/message-sent
{
  "source": "slack",
  "session": "channel:C09G4KEBYF9",
  "thread": "1784330758.592879",
  "direction": "inbound",
  "text_preview": "sanitized bounded text",
  "timestamp": "2026-07-17T23:36:51Z"
}

The bridge should keep privacy and blast radius under control: bounded fields, sanitized content, explicit event names, and read models built from the append log.

Pilot workflow

2

3

4

Cron migration

Use a low-risk recurring job as the first authority transfer.

The right pilot is a reporting or accountability loop with no automatic external side effects. Keep OpenClaw cron as the trigger while Orc learns to evaluate and produce traceable recommendations.

Keep in OpenClaw

Scheduling + delivery

Existing cron owns when things run and whether a message is sent.

Move into Orc

Evaluation + trace

Workflow logic writes recommendation events that can be inspected and tuned.
